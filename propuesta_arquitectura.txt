━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Métricas actuales de deuda técnica:
  - api.py:       2 817 líneas  → 1 archivo hace routing + lógica + persistencia
  - bot.py:       2 837 líneas  → 1 archivo es orquestador + estrategia + estado
  - index.html:   7 763 líneas  → 1 archivo es UI + lógica + estado + llamadas API
  - 29 módulos Python en el raíz plano
  - 26 tests en una sola carpeta sin jerarquía
  - Estado global mutable (dicts en módulo) en updown_learner, phantom_learner,
    phantom_optimizer, vps_experiment, risk_manager, etc.
  - Acoplamiento fuerte: bot.py importa directamente 15+ módulos distintos
  - Sin capa de abstracción de datos: JSON files accedidos directamente en cada
    módulo (15 archivos JSON distintos, cada uno con su propio open/write)

Síntomas concretos que ya ocurrieron:
  - WR inconsistente entre secciones (hubo 4 fuentes de verdad distintas)
  - Bug de substring "5m in ud15m" repetido en 3 archivos distintos
  - Parámetros con unidades incorrectas (38.0 vs 0.38) sin validación
  - Chat del cerebro respondía como bot equivocado
  - Añadir una feature (optimizer) requirió tocar api.py + bot.py + index.html
    al mismo tiempo

Consecuencia si no se refactoriza:
  Cada nueva fase duplica la superficie de bugs. Fase 10 añadió ~800 líneas
  a api.py y ~600 a bot.py. Al ritmo actual, en 3 fases más ningún archivo
  será mantenible.

2. PRINCIPIOS DE DISEÑO

2.1 Arquitectura en Capas (Layered / Clean Architecture)
  Cada capa solo conoce a la capa inmediatamente inferior. Nunca sube.

    ┌────────────────────────────────────────────┐
    │  Capa de Presentación  (API + Frontend)    │
    ├────────────────────────────────────────────┤
    │  Capa de Aplicación    (Services / Cases)  │
    ├────────────────────────────────────────────┤
    │  Capa de Dominio       (Models + Logic)    │
    ├────────────────────────────────────────────┤
    │  Capa de Infraestructura (Repos + I/O)     │
    └────────────────────────────────────────────┘

2.2 Patrones de diseño a usar
  - Repository Pattern       → abstrae toda persistencia (JSON → SQLite → DB)
  - Service Layer            → orquesta casos de uso sin tocar HTTP ni archivos
  - Event Bus (simple)       → desacopla resolución de trades de sus efectos
  - Dependency Injection     → módulos reciben dependencias, no las importan
  - Strategy Pattern         → estrategias de trading son objetos intercambiables
  - Factory                  → construcción de bots/estrategias centralizada
  - Singleton controlado     → estado compartido solo a través de repositorios

2.3 Reglas de oro
  a) Un módulo = una responsabilidad
  b) Si necesitas importar más de 3 módulos del mismo proyecto, es una señal
     de que falta una capa intermedia
  c) Toda validación de parámetros ocurre en los modelos, no en las rutas
  d) Toda persistencia ocurre en repositorios, no en servicios ni rutas
  e) Los tests no deben necesitar levantar el servidor completo

4.1 MODELOS (app/core/models/)
────────────────────────────────

Todos los modelos usan Pydantic v2. La validación de unidades vive aquí,
no en BotParams.update() ni en las rutas.

  # params.py — ejemplo de cómo se previene el bug de 38.0 vs 0.38
  class UpDownParams(BaseModel):
      min_confidence: Annotated[float, Field(ge=0.0, le=1.0)]
      momentum_gate:  Annotated[float, Field(ge=0.0, le=1.0)]

      @field_validator('min_confidence', 'momentum_gate', mode='before')
      @classmethod
      def normalize_fraction(cls, v):
          """Si viene como porcentaje entero (ej 35), divide entre 100."""
          if isinstance(v, (int, float)) and v > 1.0:
              return v / 100.0
          return v

  # stats.py
  class BotStats(BaseModel):
      bot_id:    Literal['ud5m', 'ud15m', 'ph5m', 'ph15m']
      total:     int = 0
      wins:      int = 0
      win_rate:  Optional[float] = None
      recent_wr: Optional[float] = None

      @computed_field
      @property
      def computed_win_rate(self) -> Optional[float]:
          return self.wins / self.total if self.total > 0 else None

Beneficio clave: una sola definición de "qué es un BotStats válido",
usada tanto en el backend como en la validación del frontend via OpenAPI.

4.2 REPOSITORIOS (app/infrastructure/repositories/)
──────────────────────────────────────────────────────

Abstracción completa de la persistencia. Ningún servicio sabe si los datos
viven en JSON, SQLite o PostgreSQL.

  # base.py
  class BaseRepository(ABC):
      @abstractmethod
      def load(self) -> dict: ...
      @abstractmethod
      def save(self, data: dict) -> None: ...

  # json_repo.py — implementación actual (JSON files)
  class JsonRepository(BaseRepository):
      def __init__(self, path: Path):
          self._path = path

      def load(self) -> dict:
          if self._path.exists():
              return json.loads(self._path.read_text())
          return {}

      def save(self, data: dict) -> None:
          self._path.parent.mkdir(exist_ok=True)
          self._path.write_text(json.dumps(data, indent=2))

  # stats_repo.py — repositorio específico con métodos de dominio
  class StatsRepository(JsonRepository):
      def get_phantom_wr(self, interval: Literal['5', '15']) -> Optional[float]:
          data = self.load()
          ph = data.get(interval, {}).get('phantom', {})
          total = ph.get('total', 0)
          return ph['wins'] / total if total > 0 else None

      def record_result(self, interval, won: bool, metadata: dict) -> None:
          ...

Cuando se quiera migrar a SQLite, solo se reescribe stats_repo.py.
El resto del código no cambia nada.

  # bus.py
  class EventBus:
      _handlers: dict[type, list[Callable]] = defaultdict(list)

      def subscribe(self, event_type: type, handler: Callable) -> None:
          self._handlers[event_type].append(handler)

      def publish(self, event) -> None:
          for handler in self._handlers[type(event)]:
              handler(event)

  event_bus = EventBus()  # singleton

      catch { err = { error: await resp.text() }; }
      throw new ApiError(resp.status, err);
    }
    return resp.json();
  }

  # utils/format.js — pct(), escHtml(), etc. en un solo lugar
  export const pct = v => v == null ? '—' : (v * 100).toFixed(0) + '%';
  export const escHtml = s => s.replace(/&/g,'&amp;')...

4.8 TESTING (tests/)
──────────────────────

  Tres niveles:

  UNIT (sin I/O):
    - Testean modelos, estrategias, learners, optimizer
    - Usan mocks/fakes para repos
    - Deben correr en < 1 segundo total
    - Ejemplo: test que updown_15m_min_confidence=38 → se normaliza a 0.38

  INTEGRATION (con I/O en tmp):
    - conftest.py provee tmp_path con repos reales en directorio temporal
    - Testean flujo completo: signal → trade → resolución → stats actualizados
    - Sin servidor HTTP

  API (TestClient):
    - Montan la app FastAPI con dependencias inyectadas reemplazadas por mocks
    - Testean status codes, schemas de respuesta, SSE format
    - Sin estado real de producción

  # conftest.py — ejemplo
  @pytest.fixture
  def fake_stats_repo(tmp_path):
      return StatsRepository(tmp_path / "updown_stats.json")

  @pytest.fixture
  def stats_service(fake_stats_repo):
      return StatsService(fake_stats_repo)

5. PLAN DE MIGRACIÓN (sin big bang — incremental)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

La migración se hace en fases. En cada fase el bot sigue funcionando.
Nunca hay un corte total.

FASE A — Fundamentos (no rompe nada, solo añade estructura)
  Duración estimada: 1 sesión de trabajo
  1. Crear carpeta app/ con __init__.py vacíos
  2. Crear app/core/models/ con BotParams, BotStats, Trade en Pydantic v2
  3. Crear app/infrastructure/repositories/ con JsonRepository genérico
     y StatsRepository, ParamsRepository que envuelven los JSON actuales
  4. Mover validadores de fracción de config.py a los modelos Pydantic
  5. Tests: unit/core/test_models.py verifica normalizaciones
  Resultado: nueva estructura existe en paralelo, código viejo no tocado

FASE B — Servicios (extrae lógica de api.py)
  Duración estimada: 2 sesiones
  1. Crear StatsService y que /api/bots/stats lo use
  2. Crear ParamService y que /api/params lo use
  3. Crear AdvisorService con la lógica de chat extraída de api.py
  4. api.py pierde ~800 líneas, solo quedan rutas
  5. Tests: integration/test_stats_service.py, test_param_service.py
  Resultado: api.py < 1 200 líneas, toda lógica testeable sin HTTP

FASE C — Event Bus (desacopla bot.py)
  Duración estimada: 1 sesión
  1. Crear EventBus y PhantomTradeResolved
  2. En bot.py: reemplazar los 5 imports directos en resolución phantom
     por event_bus.publish(PhantomTradeResolved(...))
  3. Cada módulo (learner, optimizer, vps) se suscribe en su __init__
  4. bot.py pierde ~400 líneas de "efectos secundarios"
  5. Tests: unit/services/test_trade_flow.py con bus de test
  Resultado: bot.py < 1 800 líneas, añadir efectos = 1 línea

FASE D — Repositories completos (desacopla persistencia)
  Duración estimada: 1 sesión
  1. Todos los open()/json.load() del proyecto → vía repositorios
  2. updown_learner, phantom_optimizer, risk_manager, vps_experiment
     reciben su repo por constructor (no hardcodean paths)
  3. Tests de integración usan tmp_path → cero contaminación de datos reales
  Resultado: migrar a SQLite luego = solo reescribir json_repo.py

FASE E — Frontend (divide index.html)
  Duración estimada: 2 sesiones
  1. Extraer utils (format.js, dom.js, api/client.js)
  2. Extraer componente bot_brain.js (el más grande y más cambiado)
  3. Extraer chat.js
  4. Extraer components restantes uno por uno
  5. index.html queda como shell de < 200 líneas
  Resultado: añadir feature UI = tocar 1 archivo JS, no index.html completo

FASE F — Runners y Orchestrator (limpia bot.py)
  Duración estimada: 1 sesión
  1. Extraer updown_runner.py, btc_runner.py, weather_runner.py de bot.py
  2. Crear orchestrator.py que los arranca/para
  3. bot.py desaparece o queda como stub de compatibilidad
  Resultado: bot.py reemplazado por runners < 400 líneas cada uno

6. NUEVAS FEATURES QUE SE DESBLOQUEAN CON ESTA ARQUITECTURA
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Con la arquitectura actual, implementar estas features costaría semanas
y generaría más deuda. Con la nueva arquitectura propuesta, costarían horas:

  1. NOTIFICACIONES (Telegram / email)
     → Nuevo handler que se suscribe a PhantomTradeResolved, RealTradeExecuted
     → No toca bot.py ni api.py

  2. MIGRACIÓN A SQLITE
     → Solo reescribir JsonRepository → SqliteRepository
     → El resto del sistema no cambia

  3. MÚLTIPLES ESTRATEGIAS EN PARALELO (A/B testing de presets)
     → StrategyRegistry permite instanciar dos UpDownStrategy con params distintos
     → OptimizerService los compara por BotStats.win_rate

  4. AUTENTICACIÓN / MULTI-USUARIO
     → Middleware de auth en app/api/middleware/auth.py
     → Los servicios reciben user_id, los repos particionan por usuario

  5. API VERSIONING
     → /api/v1/ y /api/v2/ como routers separados
     → Servicios compartidos entre versiones

  6. MÉTRICAS / OBSERVABILIDAD
     → Middleware que mide duración de requests y la publica al EventBus
     → Sin tocar lógica de negocio

7. RESUMEN EJECUTIVO
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  Problema central:
    La plataforma creció 9 fases con arquitectura "flat file". Cada feature
    nueva toca los mismos 3 archivos gigantes. Los bugs se repiten porque
    la misma lógica existe en 3 lugares distintos.

  Solución:
    Arquitectura en capas (Clean Architecture light) con:
    - Modelos Pydantic v2 como única fuente de verdad de validación
    - Repositorios que abstraen toda persistencia
    - Servicios que concentran la lógica de negocio
    - Event Bus que desacopla efectos secundarios
    - Rutas HTTP que solo parsean y delegan
    - Frontend dividido en componentes JS reutilizables
    - Tests en 3 niveles (unit / integration / api)

  Método de migración:
    Incremental en 6 fases (A→F). El bot nunca se detiene.
    Cada fase produce valor inmediato y tests que lo verifican.
    Sin big bang. Sin riesgo de romper producción.

  Archivos actuales → nuevo destino:
    api.py         → app/api/routes/* + app/services/*
    bot.py         → app/infrastructure/bot_runner/* + eventos
    config.py      → app/core/models/params.py + app/infrastructure/repositories/
    updown_learner → app/core/learners/ + app/infrastructure/repositories/
    phantom_opt.   → app/core/optimizer/ + app/services/optimizer_service.py
    index.html     → frontend/index.html (shell) + frontend/js/components/*
    tests/         → tests/unit/ + tests/integration/ + tests/api/
